# OpenPrints Nostr Event Schema

## 1. Title & Overview

OpenPrints publishes 3D-printing design metadata as Nostr events. It uses **parameterized replaceable events** so each design can be updated over time (e.g. fixing metadata or linking a new file). This document defines the small set of kinds and tag schema used by OpenPrints. It is a canonical “mini-NIP” for the project, not a general-purpose Nostr NIP.

---

## 2. Background & Design Goals

OpenPrints relies on **NIP-33** (Parameterized Replaceable Events). We do not assume that Nostr kind numbers are globally reserved; safety and correctness come from the tuple `(pubkey, kind, d)` and from the tag schema we require.

Design goals:

- **Stable identity** — Each design has a single logical identity per creator.
- **Replaceability** — Newest version wins for a given `(pubkey, kind, d)`.
- **Integrity** — File hashes (e.g. `sha256`) allow clients and indexers to verify file content.
- **Composability** — Endorsements (“I printed this”) and Lightning zaps (NIP-57) attach to designs without changing the core design event kind.

The uniqueness of a design is defined by **`(creator_pubkey, kind, d)`**. We keep app-specific semantics in tags and in this document rather than relying on kind numbers alone. Indexers and clients should validate events by kind and by the presence and shape of the required tags.

---

## 3. Kind 33301 — OpenPrints Design Listing (Parameterized Replaceable)

### 3.1 Kind number and semantics

- **Kind:** `33301`
- **Range:** In the 30_000–39_999 range used for parameterized replaceable events (NIP-33). The “3” digits are a naming choice only; they have no protocol effect.
- **Semantics:** One 3D design listing from a single creator. One event = one current version of that design.

**Replaceability:** This is a **parameterized replaceable event**. The tuple `(creator_pubkey, kind=33301, d)` identifies one logical design. Among events with the same `(pubkey, kind, d)`, the event with the highest `created_at` is the current version; older events may be kept for history (indexer-dependent).

### 3.2 Identity via `d` tag

`d` is the **stable design identity** for that creator. It is **not** the file hash.

- `d` is **generated by the client** when the creator first publishes the design, not typed by the user.
- We recommend a UUID-based scheme with a namespace prefix so that:
  - the same design can be updated in place across multiple replacement events, and
  - multiple apps used by the same creator do not collide, even if they follow the same tag scheme.

Recommended format:

```text
["d", "openprints:<uuid-v4>"]
```

Example:

```text
["d", "openprints:3f17122b-6547-42db-a9ac-d76a61c5e1cc"]
```

The UUID is:

- generated once when the creator first publishes the design, and  
- reused for all subsequent updates of that design.

**Why we do not use `sha256(file)` as `d`:** If `d` were the file hash, every change to the file would create a new identity and break replaceability. We still use file hashes for integrity, but in a separate **`sha256`** tag.

### 3.3 Required tags for kind 33301

| Tag    | Description |
|--------|--------------|
| `d`    | Stable design identity for that creator. Recommended format: `openprints:<uuid-v4>`. |
| `name` | Human-readable name of the design. |
| `format` | Primary file format: `stl`, `3mf`, or other supported extension. |
| `sha256` | SHA-256 hash of the main printable file (hex). Used to verify file integrity. |
| `url`  | HTTPS URL to the main file (e.g. on R2/S3/Blossom or other storage). |

`name` validation rules (for indexers and clients):

- Purpose: display title for humans; not part of stable identity (`d` is identity).
- Length: **1 to 120 characters** after trimming.
- Character set: Unicode text is allowed (including accents and non-ASCII letters).
- Disallowed: control characters, null bytes, and invisible/bidi control characters.
- Whitespace: trim leading/trailing whitespace; collapse repeated internal whitespace for indexing/display normalization.
- Normalization: normalize to Unicode NFC for storage/comparison.
- Uniqueness: not required to be unique.

Example tag set (conceptually):

```text
["d", "openprints:3f17122b-6547-42db-a9ac-d76a61c5e1cc"]
["name", "Phone Stand"]
["format", "stl"]
["sha256", "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"]
["url", "https://example.com/files/phone-stand.stl"]
```

### 3.4 Recommended tags for kind 33301

These are not required but strongly encouraged for discovery and UX:

| Tag       | Description | Multiplicity |
|-----------|-------------|--------------|
| `preview` | HTTPS URL to a preview image. | Multiple allowed |
| `category`| Category hint, e.g. `holder`, `tool`, `toy`. | Multiple allowed |
| `material`| Materials tested by the maker for this design, e.g. `PLA`, `PETG`, `ABS`. | Multiple allowed |
| `printer` | Printers tested by the maker for this design, e.g. `Prusa MK4`, `Bambu P1P`. | Multiple allowed |
| `license` | License identifier, e.g. `CC-BY-4.0`, `MIT`, `proprietary`. | Single recommended |
| `lnurl`   | LNURL pay address for the creator (Lightning tips). | Single recommended |
| `m`       | MIME type of the main file, e.g. `model/stl`. | Single recommended |

Optional: **`version`** — string or integer to track logical version numbers alongside `created_at`. Indexers may use this for display or ordering; replaceability is still defined by `created_at`.

Multiplicity rules:

- Repeated tags are represented as repeated tag arrays (for example, multiple `category` tags), not comma-separated values in a single tag.
- For repeated `preview` tags, clients should treat the first `preview` as the primary thumbnail and remaining previews as secondary images.

An indexer should interpret these tags to support search, filtering, and display (e.g. by category, tested materials/printers, and license).

### 3.5 Event `content` (design description)

For kind `33301`, `content` is optional free-form text intended for a human-readable design description.

- `content` **may be empty**.
- `content` **may contain Markdown** (for example: headings, lists, links, and code fences).
- Structured fields (`d`, `name`, `format`, `sha256`, `url`, etc.) remain canonical in tags; clients and indexers should not rely on parsing `content` for required machine data.

Indexers should store `content` as untrusted text, and clients should sanitize before rendering Markdown in the UI.

### 3.6 Replaceability rules

- One logical design = `(creator_pubkey, kind=33301, d="openprints:<uuid>")`.
- If multiple events exist with the same tuple, the event with the **highest `created_at`** is the current version.
- Older events may be stored for history at the indexer’s discretion.

## 4. Kind 33311 — Endorsement (“I printed this”)

### 4.1 Kind number and semantics

- **Kind:** `33311`
- **Semantics:** Version-scoped endorsement state from a user for a specific design event (kind `33301`). The **author** is the user who printed.

This kind is modeled as **parameterized replaceable state** per `(endorser_pubkey, kind=33311, d)` where:

- `d` is deterministic for the targeted design (derived from the `e` tag).
- latest `created_at` wins for that `(pubkey, kind, d)`.
- this allows users to toggle endorsement on/off (`endorsed: 1` or `0`) for each design version.

OpenPrints allows a user to endorse multiple versions of the same logical design. Indexers should then aggregate:

- **Version-level:** per `e` (design event id).
- **Design-level (user deduped):** if a user has `endorsed=1` on one or more versions of the same logical design, count as **1 unique user endorsement** for that design.

### 4.2 Required tags for kind 33311

| Tag     | Description |
|---------|-------------|
| `d`     | Deterministic endorsement key for the targeted design, e.g. `openprints:endorse:<design_event_id>`. |
| `e`     | Event id of the design version (kind `33301`) being endorsed. |
| `endorsed` | Endorsement state as string: `\"1\"` (endorsed) or `\"0\"` (not endorsed / removed). |

### 4.3 Optional fields

- `rating` as `"1"` … `"5"`.
- `material` and `printer` used for the print.
- Event `content` as free-text review/comment.
- Future tags may include nozzle size, layer height, filament brand/color, etc. Indexers may ignore unknown tags or use them when defined.

## 5. Zap Receipts (Lightning, NIP-57)

OpenPrints does **not** define a new kind for Lightning payments. It uses existing **NIP-57** zap receipts (kind **9735**).

When zap receipts reference a design event (via their tags), an indexer may:

- show total sats received per design,
- list recent tippers or zap activity.

No additional OpenPrints-specific zap semantics are required beyond NIP-57.

## 6. Future Extensions (Optional Kinds)

The following are **placeholders** only; they are not yet defined:

- **33321** — Design collections / catalogs.
- **33331** — Parametric or generator metadata.

Details (tags, replaceability) will be specified in a later revision or NIP if adopted.

## 7. Identity, Namespaces & Collisions

Event kinds are not globally reserved in Nostr. Other applications could use 33301 or 33311 for different purposes. OpenPrints avoids collisions and misattribution by:

- **Creator pubkey** — Each creator has their own namespace of designs.
- **`d` semantics** — UUID-based, client-generated, not human-typed; reduces collisions and preserves replaceability.
- **Tag schema** — We require specific tags (`sha256`, `url`, `format`, `name`, etc.). Events that do not match the expected schema can be ignored.

An indexer should:

- filter events by kind and by presence/format of required tags, and  
- ignore events that do not match the OpenPrints schema.

In practice, misuse of the same kind by another app will not break OpenPrints clients or indexers that validate by tag schema.

### 8.3 Example zap receipt (kind 9735, NIP-57)

Zap receipts are produced by the recipient’s Lightning service when an invoice is paid. The `e` tag references the design event being zapped so an indexer can aggregate sats per design. `p` is the recipient (design creator); `bolt11` and `description` are required by NIP-57.

```json

  "created_at": 1730000200,
  "kind": 9735,
  "tags": [
    ["p", "b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef12345678"],
    ["e", "a1b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef123456"],
    ["bolt11", "lnbc100n1..."],
    ["description", "{\"kind\":9734,\"tags\":[[\"e\",\"a1b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef123456\"],[\"p\",\"b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef12345678\"]],\"content\":\"\",...}"]

  "sig": "e3f4a5b6c7d8e9f0a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2"
```
